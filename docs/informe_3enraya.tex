\section{Introduccion}

3EnRaya es un sistema cliente-servidor desarrollado en lenguaje C para Linux.
El servidor central recibe conexiones TCP, registra jugadores, mantiene una
lista dinamica de participantes, administra una cola de espera y controla
completamente una partida de tres en raya 1 vs 1.

El juego no se ejecuta entre pares directamente. Los clientes solo envian
comandos al servidor y reciben respuestas. El servidor valida turnos,
movimientos, tablero, ganador, empate, abandono y marcador. Esta decision
permite que exista un unico estado oficial del juego.

\section{Objetivo general}

Disenar e implementar un protocolo de aplicacion y un servicio de red que
permita a varios jugadores participar en partidas de tres en raya mediante un
servidor central TCP.

\section{Objetivos especificos}

\begin{itemize}
    \item Implementar un servidor central en C.
    \item Permitir multiples clientes conectados al mismo tiempo.
    \item Usar \texttt{select(2)} para multiplexar sockets sin usar
    \texttt{fork(2)} ni \texttt{pthread}.
    \item Usar estructuras dinamicas para almacenar jugadores y cola de espera.
    \item Definir comandos y respuestas en un protocolo de texto.
    \item Administrar el juego completamente desde el servidor.
    \item Registrar participantes y marcador de forma efimera mientras el
    servidor se mantiene en ejecucion.
    \item Documentar el servicio con formato similar a RFC.
\end{itemize}

\section{Descripcion del servicio}

El servicio permite que varios jugadores se conecten a un servidor central.
Cada jugador inicia una sesion interactiva desde la aplicacion cliente. Al
conectarse, el jugador recibe un mensaje de bienvenida y debe registrarse con
el comando \texttt{HELLO <nombre>}.

Despues del registro, el jugador queda en estado \texttt{LOBBY}. Desde ese
estado puede solicitar el listado de participantes, consultar marcador, pedir
tablero si esta en partida, entrar a cola o salir. Cuando dos jugadores entran
a la cola, el servidor los empareja y crea una partida.

Durante la partida, el servidor administra el intercambio de mensajes y valida
cada jugada. Los clientes no tienen autoridad para modificar el tablero por su
cuenta; solo solicitan movimientos usando \texttt{MOVE <1-9>}.

\section{Makefile}

El proyecto usa un Makefile inspirado en el formato de clase. Se conservan
variables como \texttt{EXT}, \texttt{SRCDIR} y \texttt{OBJDIR}. La diferencia
es que este proyecto genera dos ejecutables, porque servidor y cliente tienen
funciones \texttt{main} independientes.

\begin{lstlisting}[style=code]
CC = gcc
CFLAGS = -std=c11 -Wall -Wextra -Wrestrict -pedantic -O2
EXT = .c
SRCDIR = src
OBJDIR = obj
BINDIR = bin
SERVER_APP = $(BINDIR)/3enraya_server
CLIENT_APP = $(BINDIR)/3enraya_client
\end{lstlisting}

El objetivo \texttt{all} construye:

\begin{lstlisting}[style=code]
bin/3enraya_server
bin/3enraya_client
\end{lstlisting}

El objetivo \texttt{test} ejecuta una prueba automatizada con dos clientes.

\section{Protocolo 3ENRAYA/1.0}

El protocolo es de texto sobre TCP. Cada comando y cada respuesta termina con
salto de linea \texttt{\textbackslash n}. Esto facilita la lectura, prueba y
depuracion desde terminal.

\section{Logica del juego}

El juego es tres en raya. El tablero tiene nueve casillas numeradas del 1 al 9:

\begin{lstlisting}[style=code]
1 | 2 | 3
---------
4 | 5 | 6
---------
7 | 8 | 9
\end{lstlisting}

Reglas:

\begin{itemize}
    \item Cada partida tiene dos jugadores.
    \item El primer jugador emparejado usa \texttt{X}.
    \item El segundo jugador emparejado usa \texttt{O}.
    \item Siempre inicia \texttt{X}.
    \item Solo puede mover el jugador indicado por \texttt{TURN}.
    \item Un movimiento valido usa una casilla libre entre 1 y 9.
    \item Gana quien completa fila, columna o diagonal.
    \item Si se llenan las nueve casillas sin ganador, hay empate.
\end{itemize}

\section{Estados del jugador}

\begin{lstlisting}[style=code]
DESCONECTADO -> CONECTADO -> LOBBY -> EN_COLA -> EN_PARTIDA -> LOBBY
\end{lstlisting}

Desde cualquier estado conectado, el jugador puede pasar a
\texttt{DESCONECTADO} por \texttt{QUIT}, error o desconexion.

\section{Gestion de varios clientes}

El servidor usa una lista enlazada para jugadores conectados. Cada conexion
aceptada crea un nuevo nodo \texttt{Player}. El servidor no usa un arreglo fijo
para almacenar jugadores conectados.

La cola de espera se implementa con \texttt{queue\_head} y
\texttt{queue\_tail}. Cuando hay dos jugadores en cola, el servidor crea una
partida con \texttt{create\_match}.

La multiplexacion se realiza con \texttt{select(2)}. El ciclo principal agrega
al conjunto de lectura el socket del servidor y los sockets de todos los
clientes. Si hay actividad en el socket del servidor, se acepta un cliente. Si
hay actividad en un cliente, se lee su comando.

\section{Procesamiento de mensajes}

\subsection{Servidor}

El servidor recibe bytes con \texttt{recv}. Como TCP es flujo de bytes, usa el
buffer del jugador para acumular datos hasta encontrar \texttt{\textbackslash n}.
Luego procesa la linea completa con \texttt{handle\_command}.

Flujo:

\begin{enumerate}
    \item \texttt{receive\_from\_player} recibe bytes.
    \item \texttt{process\_player\_line} limpia la linea.
    \item \texttt{handle\_command} interpreta el comando.
    \item Se llama a la funcion correspondiente.
    \item \texttt{send\_line} construye y envia la respuesta.
\end{enumerate}

El servidor muestra el intercambio en consola:

\begin{lstlisting}[style=code]
[RX <- Ramiro] MOVE 5
[TX -> Paulo] TURN O Paulo
\end{lstlisting}

\subsection{Cliente}

El cliente usa \texttt{select} para atender teclado y socket. Si llega un
mensaje del servidor, lo interpreta con \texttt{print\_server\_message}. Si el
usuario escribe un comando, lo envia al servidor con \texttt{send\_all}.

\section{Demostracion}

La cartilla pide iniciar servidor y por lo menos tres clientes. La demostracion
recomendada usa cuatro terminales.

\subsection{Terminal 1: servidor}

\begin{lstlisting}[style=code]
make clean
make
./bin/3enraya_server --host 127.0.0.1 --port 5000
\end{lstlisting}

\subsection{Terminales 2, 3 y 4: clientes}

\begin{lstlisting}[style=code]
./bin/3enraya_client --host 127.0.0.1 --port 5000 --name Ramiro
./bin/3enraya_client --host 127.0.0.1 --port 5000 --name Paulo
./bin/3enraya_client --host 127.0.0.1 --port 5000 --name Carla
\end{lstlisting}

Pasos:

\begin{enumerate}
    \item En Carla escribir \texttt{PLAYERS} para mostrar tres jugadores.
    \item En Ramiro y Paulo escribir \texttt{QUEUE}.
    \item Mostrar que Carla sigue en \texttt{LOBBY} y puede consultar
    \texttt{SCORE} o \texttt{PLAYERS}.
    \item Probar un movimiento fuera de turno.
    \item Completar una partida.
    \item Mostrar \texttt{RESULT WIN} o \texttt{RESULT DRAW}.
    \item Mostrar el marcador final.
\end{enumerate}

\section{IA y conclusiones}

\subsection{Herramienta de IA utilizada}

Se utilizo asistencia de IA como apoyo para revisar requisitos, organizar la
documentacion, preparar explicaciones y detectar puntos debiles frente a la
cartilla. Esta seccion debe completarse con el nombre exacto de la herramienta
utilizada y si fue version gratuita o de pago.

\subsection{Evaluacion de los aportes de IA}

Los aportes fueron utiles para ordenar el protocolo, identificar funciones
importantes y preparar una guia de defensa. Sin embargo, fue necesario revisar
el codigo, contrastarlo con los requisitos y comprobar que no se usaran
\texttt{fork} ni \texttt{pthread}. La IA funciono como apoyo, no como reemplazo
de la comprension del proyecto.

\subsection{Conclusiones}

\begin{itemize}
    \item El servidor central controla completamente el juego.
    \item \texttt{select} permite atender multiples clientes sin usar procesos
    ni hilos.
    \item La lista enlazada de jugadores evita un limite fijo en la estructura
    principal de conexiones.
    \item La cola permite emparejar jugadores de forma ordenada.
    \item El protocolo de texto facilita la prueba, depuracion y defensa.
    \item El registro y marcador son efimeros, como permite la cartilla.
\end{itemize}

\subsection{Aprendizajes}

\begin{itemize}
    \item Uso de sockets TCP en C.
    \item Diferencia entre \texttt{127.0.0.1} y \texttt{0.0.0.0}.
    \item Multiplexacion con \texttt{select}.
    \item Diseno de protocolos de aplicacion.
    \item Separacion entre cliente, servidor, comandos y respuestas.
    \item Importancia de documentar estructuras, campos y funciones.
\end{itemize}
